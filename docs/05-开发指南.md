# 开发指南

本文面向项目维护者和插件开发者，说明如何在当前代码结构下开发、调试和扩展 Consilens。

## 一、开发环境

基础要求：

- JDK 11
- Maven 3.8+

推荐构建参数：

```bash
MAVEN_OPTS="-Xmx8g -Xms8g -DsocksProxyHost= -DsocksProxyPort= -Djava.net.useSystemProxies=false"
```

常用命令：

```bash
mvn clean package -DskipTests -Prelease
mvn test
mvn -pl consilens-core test
mvn verify
```

## 二、模块职责

开发前先确认修改点属于哪个模块：

- CLI / 配置 / 输出：`consilens-cli`
- 比对规划 / 算法 / 分段 / 差异模型：`consilens-core`
- 连接器 / 数据集抽象：`consilens-connector`
- Sink 输出插件：`consilens-sink`
- Plugin 运行时：`consilens-spi`
- 发行包：`consilens-dist`

### 模块依赖规则

- `consilens-core` 依赖 `consilens-connector-api`（不依赖具体连接器插件）
- `consilens-connector-base` 依赖 `consilens-connector-api`
- 连接器插件模块不得依赖 `consilens-core`
- `consilens-spi` 是通用插件基础设施，不属于 connector 语义

## 三、代码风格

项目默认约束：

- Java 11
- 四空格缩进
- 类名 `UpperCamelCase`
- 方法和字段 `lowerCamelCase`
- 常量 `SCREAMING_SNAKE_CASE`
- 引入类时使用 `import`，不要写全限定名

## 四、配置开发注意事项

配置入口是：

- [CliConfiguration.java](../consilens-cli/src/main/java/com/consilens/cli/model/CliConfiguration.java)

配置模型使用 connector/dataset 结构：

- `source/target`：各含 `type`、`name`、`connection`（连接参数）、`resource`（数据集标识）
- `comparison`：`keys`、`comparisons`（原 `compareColumns`）、`filters`（原 `where`）
- `strategy`、`concurrency`、`normalization`、`result`

变更配置字段时，通常需要同时检查：

- `CliConfiguration`
- `ConfigurationManager`
- `ConfigNormalizer`
- `ConnectorConfigMapper`
- `DiffService`
- `README` 与 `docs`

建议：

- 新配置必须有默认值或兼容旧行为
- 文档要同步更新
- 示例配置要与当前实现一致

## 五、核心算法开发

### `checksum` 路径

主要类：

- [ChecksumDiffer.java](../consilens-core/src/main/java/com/consilens/core/algorithm/ChecksumDiffer.java)
- [TableSegmenter.java](../consilens-core/src/main/java/com/consilens/core/segment/TableSegmenter.java)
- [TableSegment.java](../consilens-core/src/main/java/com/consilens/core/segment/TableSegment.java)

修改这条链路时，重点关注：

- 分段边界是否有缺口或重叠
- checksum 相等时是否正确跳过
- 小段是否能及时收敛到本地比较
- `strategy.localCompare.mode` 是否按配置切到 `row-hash` 或 `full`
- `infoTree` 是否同步记录了新决策

### `join` 路径

主要类：

- [JoinDiffer.java](../consilens-core/src/main/java/com/consilens/core/algorithm/JoinDiffer.java)

修改重点：

- JOIN SQL 的正确性
- 同库校验逻辑
- 差异行的主键提取与 changed columns 输出

### 比对规划器

- [DefaultComparePlanner.java](../consilens-core/src/main/java/com/consilens/core/compare/DefaultComparePlanner.java)

规划器只基于 `CapabilitySet` 选择执行计划，不检查连接器类型。修改时确保不引入类型判断逻辑。

## 六、如何新增执行计划

当需要新增比对策略（如 `streaming` 或自定义策略）时：

### 1. 定义 PlanExecutor

实现 `PlanExecutor` 接口，放在 `consilens-core` 的 `compare/executor/` 包下：

```java
public class StreamingPlanExecutor implements PlanExecutor {
    @Override
    public CompareResult execute(ComparePlan plan, CompareExecutionSettings settings) {
        // ...
    }
}
```

### 2. 注册到 DefaultComparePlanner

在 `DefaultComparePlanner.buildPlan()` 中，基于 `CapabilitySet` 增加新分支：

```java
if (source.capabilities().has(Capability.STREAMING_READ)
        && target.capabilities().has(Capability.STREAMING_READ)) {
    return new StreamingComparePlan(...);
}
```

**注意：不要在规划器中检查连接器类型（如 instanceof）**，只允许通过 `CapabilitySet` 来决策。

### 3. 在 CapabilitySet 中声明能力

若需要新的能力，在 `Capability` 枚举中新增条目，并在对应 `DatasetHandle` 实现中声明。

## 七、如何新增 Sink 插件

### 1. 新建模块

建议在：

- `consilens-sink/consilens-sink-plugins/`

下新增独立模块，命名规则：`consilens-sink-<format>` 或 `consilens-sink-<format>-<type>`。

### 2. 实现 SinkProvider

实现 `SinkProvider` 接口，指定 `format` 和 `type`：

```java
public class MyCustomSinkProvider implements SinkProvider {

    @Override
    public String getFormat() {
        return "custom";
    }

    @Override
    public String getType() {
        return "diff-record";
    }

    @Override
    public Sink create() {
        return new MyCustomSink();
    }
}
```

### 3. 实现 Sink

实现 `Sink` 接口，按需覆盖所需回调：

```java
public class MyCustomSink implements Sink {

    @Override
    public void open(SinkConfig config, DiffContext context) throws Exception {
        // 解析 properties，初始化资源
        MyConfig cfg = new ObjectMapper().readValue(config.getProperties(), MyConfig.class);
    }

    @Override
    public void onDiffRecords(List<DiffRow> rows, DiffContext context) throws Exception {
        // 处理差异明细
    }

    @Override
    public void close() throws Exception {
        // 释放资源
    }
}
```

### 4. 注册 SPI

在资源目录新增：

`META-INF/services/com.consilens.sink.api.SinkProvider`

内容：

```text
com.example.MyCustomSinkProvider
```

### 5. 纳入打包

确保：

- 父 POM 引用了模块
- `consilens-sink-all` 引入了该依赖（如需内置）或单独作为插件放入 `plugins/` 目录

## 八、如何新增连接器

### 新增 JDBC 数据库连接器

#### 1. 新建模块

建议在：

- `consilens-connector/consilens-connector-plugins/`

下新增独立模块。

#### 2. 实现方言

建议继承：

- [AbstractDatabaseDialect.java](../consilens-connector/consilens-connector-plugins/consilens-connector-base/src/main/java/com/consilens/conncetor/base/AbstractDatabaseDialect.java)

至少补齐：

- `getDatabaseType()`
- `CapabilityProvider`
- `SqlQueryGenerator`
- `MetadataQueryGenerator`
- `DataTypeHandler`

#### 3. 注册 SPI

在资源目录新增：

`META-INF/services/com.consilens.connector.api.DatabaseDialectProvider`

内容：

```text
com.example.YourDatabaseDialectProvider
```

同时在 API 模块实现对应的 `DatabaseDialectProvider`。

#### 4. 纳入打包

确保：

- 父 POM 引用了模块
- `consilens-dist` 打包时能把连接器带进 `plugins/`

### 新增非 JDBC 数据源连接器

对于 Elasticsearch、MongoDB、HDFS 等非 JDBC 数据源：

1. 实现 `ConnectorProvider`，在 `createDatasetHandle()` 中返回自定义 `DatasetHandle`
2. `DatasetHandle` 通过 `getCapabilities()` 声明能力（如 `SERVER_SIDE_HASH`）
3. 实现 `DatasetHandle.getSupport(Class<T>)` 提供必要的执行支持接口
4. 注册 SPI 并纳入打包

核心规划器 `DefaultComparePlanner` 只基于 `CapabilitySet` 选择计划，不区分数据源类型，因此新连接器无需修改核心代码。

## 九、测试建议

### 单元测试

优先覆盖：

- 边界分段
- checksum 决策分支
- 能力集合与规划器选择
- 差异类型生成
- 输出格式转换

已有测试位置：

- [consilens-core/src/test/java](../consilens-core/src/test/java)
- [consilens-cli/src/test/java](../consilens-cli/src/test/java)

### 集成测试

适合覆盖：

- 不同数据库方言 SQL
- 类型标准化差异
- row-hash 本地比较在真实库上的行为

## 十、文档维护规则

凡是以下内容变化，都应同步更新文档：

- 配置字段
- 支持的输出格式
- 支持的连接器列表
- 算法阈值语义
- 打包产物结构

最常见错误是：

- 代码已经改成新配置模型，README 还保留旧 YAML 结构

## 十一、调试建议

### 看执行树

优先检查 `infoTree`：

- 哪一层开始出现差异
- 是继续分段还是转本地比较
- 哪些段被 checksum 直接跳过

### 看日志

默认日志配置在：

- [log4j2.xml](../consilens-core/src/main/resources/log4j2.xml)

启动脚本会把日志写到：

- `logs/`

### 看输出结果

优先确认：

- 主键提取是否正确
- 列名顺序是否一致
- 标准化配置是否影响值比较

## 十二、提交流程建议

建议遵循：

1. 先改代码
2. 再补测试
3. 最后更新文档
4. 本地至少执行一次 `mvn test` 或最小范围模块测试

提交信息建议使用 Conventional Commits：

- `feat:`
- `fix:`
- `refactor:`
- `docs:`

## 十三、已知实现边界

开发时要注意这些不是"文档问题"，而是当前代码真实边界：

- `local` 策略未实现
- `--test-connection` 未真正执行连接测试
- 数据库枚举与实际连接器列表并不完全一致
- 文档应以代码和打包产物为准，而不是营销文案
